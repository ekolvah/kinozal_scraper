# Information architecture — где живёт знание процесса

Конституция информационной архитектуры (IA) агентного процесса этого репо.
Отвечает на вопрос **«какой факт в каком файле должен жить и почему»** — в
отличие от [`files-map.md`](files-map.md), который лишь индексирует «какой файл
на какой вопрос отвечает» (одна строка на файл). Здесь — *модель и правила*;
там — *индекс*.

Источник истины по самим принципам разработки и workflow остаётся
[`principles.md`](principles.md); этот документ governs **раскладку** знания по
носителям, не его содержание.

## 1. Tier-модель носителей знания (официальная, Claude Code)

Claude Code задаёт не имена `docs/*` (их стандарт не регламентирует — переименовывать
не к чему), а **иерархию носителей знания**. Целевая раскладка:

| Tier | Назначение | Когда загружается |
|---|---|---|
| `CLAUDE.md` (root) | Тонкий роутер: что за app + env-граблии + указатели на остальное. **Цель < 200 строк** | каждую сессию, целиком |
| `.claude/rules/*.md` | Операционные инструкции, **один файл = одна тема**; можно path-scoped через frontmatter `paths:` | каждую сессию (или только при работе с matching-путями) |
| `.claude/commands/`, `.claude/agents/`, `.claude/settings*.json` | Команды (`/plan`,`/implement`) / сабагенты / permissions-deny | по вызову / при старте |
| `docs/architecture/*.md` | Reference: как устроен код (runtime/pipeline/storage/gemini/test-coverage/ci) + этот IA-doc + `principles.md` | по требованию (агент читает при необходимости) |
| `~/.claude/projects/<repo>/memory/` | Auto-memory: **только машинно/процессно-специфичное** (см. §4) | `MEMORY.md` индекс — каждую сессию |

**Честно про токены.** Файлы в `.claude/rules/` *без* `paths:` грузятся каждую
сессию ровно как `CLAUDE.md` — это **не** «меньше токенов сходу». Выигрыш в:
(a) **дедупе** — правило живёт в одном месте, а не `CLAUDE.md` ≈ `principles.md`;
(b) **single-responsibility** — одна тема на файл; (c) **path-scoping** — файл с
`paths: [tests/**]` не грузится, когда тесты не трогаем (единственный
токен-позитивный случай).

## 2. Canonical-home правило (источник истины раскладки)

> **У каждого факта — ровно один дом. Прочие упоминания — только ссылка, никогда не перефраз.**

Конкретизация (crisp, чтобы правило было исполнимо, а не лозунгом):

- **Операционные процедурные правила** (workflow: ветка / no-main-push / no-self-merge /
  one-PR / labels / plan→implement) → переносятся в `.claude/rules/` **целиком —
  правило и его rationale вместе**, не расщепляются. Старое место → строка-указатель.
  *Почему целиком:* расщепление «правило тут, rationale там» само воссоздаёт дубль.
- **Формулировки принципов §I–VI** → остаются каноном в [`principles.md`](principles.md).
  На них ссылаются по номеру (`architect-reviewer.md`, `implement.md`); **нумерацию не трогать**.
- **Энфорс-факты** (git-запреты) → канон = `.claude/settings.json` `permissions.deny`
  (+ синхрон-тест `tests/test_settings_deny.py`). **Mirror-файлов в `rules/` не создавать** —
  это дубль по определению (переоткрыл бы закрытый дубль #9) и лишний always-load контекст.
- `.claude/rules/`-файл **не должен** содержать перефраз принципа из `principles.md`
  или строки deny — только ссылку либо процедуру, которой больше нигде нет.

**Граница энфорсится человеком на ревью.** `grep` ловит лишь дословные копии, не
семантический перефраз; поэтому при переносе правила ревьюер проверяет, что в старом
месте осталась **ссылка, а не пересказ**. Скрипт-детектор семантических дублей сознательно
**не строим** — он дал бы ложное чувство покрытия (нарушение §IV Visibility Over Silence:
зелёный детектор, пропускающий перефраз, хуже честного «проверяет человек»).

## 4. Memory ↔ repo: resolved-policy

**Проектные инструкции живут в репозитории** (`.claude/`, `docs/`, скрипты, шаблоны),
а не в приватной out-of-repo Claude-памяти. Out-of-repo память — **только** для
машинно/окружение-специфичного или для стиля работы с конкретным оператором; при клоне
репо на другой машине проектное знание из приватной памяти не видно → источник истины
расщепляется.

Это **действующая политика, не открытый backlog** (N1): персона `architect-review`
раньше держалась на out-of-repo памяти, её перенесли в репо
(`.claude/agents/architect-reviewer.md` + гейт `validate_issue_sections.py` +
`principles.md §Governance`), а память удалили (#150). Соответствует принципу
«скрипты > инструкции»: знание должно быть воспроизводимым и видимым, а не приватной заметкой.

## 5. Сознательно НЕ делаем

- **Не переименовываем** `docs/architecture/*` — стандарта на их имена нет, churn
  рвёт git-историю и ссылки.
- **Не двигаем** `principles.md` — 10+ ссылок; остаётся каноническим reference/конституцией.
- **Не сливаем** reorg + behavioral + docs в один PR (one-PR-one-unit; precedent #39→#40).
- **Не строим** скрипт-детектор семантических дублей (ложное покрытие — §IV).
- **Не создаём mirror-файлы** в `rules/` для фактов, чей дом — `principles.md` или
  `settings.json` (это и есть дубль по определению).
