# Token baseline — расход токенов агентной разработки

**На какой вопрос отвечает этот файл:** сколько токенов dev-time (не рантайм пайплайнов) тратит
агентная разработка этого репо, разложенное по категориям, и какие точечные оптимизации этим
оправданы. Это **baseline** (приоритет (2) цель-функции, §V «root cause before fix» применённый к
токенам): измеряем прежде чем оптимизировать, чтобы не строить инфраструктуру спекулятивно (#196).

> **Зачем именно baseline, а не сразу фиксы.** «Честно про токены» в
> [`project-map.md`](project-map.md#честно-про-токены) уже фиксирует, что path-scoping — единственный
> токен-позитивный рычаг по always-load. Этот док проверяет числом, **окупается ли** хоть один из
> предложенных рычагов — и показывает, что главный кандидат (`mcp-on-demand`) уже реализован
> harness'ом, т.е. baseline **предотвратил** work-for-work.

## Метод и выборка (воспроизводимость)

- **Дата замера:** 2026-06-17.
- **Инструменты:**
  - `npx ccusage@20.0.14` (`daily` / `session --json`) — агрегатор реального расхода по сессиям/моделям;
  - `tiktoken` (`o200k_base`) — **прокси-токенайзер** для детерминированного счёта файлов/блоков.
    Точный токенайзер Claude офлайн недоступен (нет `ANTHROPIC_API_KEY`, `count_tokens` требует сети);
    `o200k_base` — реальный BPE (не «символы ≈ токены»), расхождение с Claude обычно в пределах ~10–15 %.
    Числа ниже читать как «порядок величины», не как точную сверку до токена.
- **Выборка для агрегатора:** все активные дни окна **2026-06-10 … 2026-06-17** (6 активных дней).
  Суммарно по `ccusage daily`: **127 447 719** токенов (вкл. cache-read), **$133.22**.
  Повторный `ccusage daily --since 20260610 --until 20260617` по той же выборке обязан давать
  сверяемое число (даты сессий неизменны).
- **subagent-спавн** замерен на реальном прогоне `architect-reviewer` в сессии планирования #196
  (`ccusage`/Task usage сообщил расход отдельной Task-сессии).

## Baseline по категориям

Категории разнесены **по методу замера** — ни `/cost`, ни `ccusage`, ни OTEL не дают разбивку
«always-load vs file-reads vs …» внутри одной main-сессии (они агрегируют по модели/сессии и
input/output/cache). Поэтому:

### A. Детерминированно измеримые токенайзером (точное число, без агрегатора)

| Категория | Токены (`o200k_base`) | Замечание |
|---|---:|---|
| **always-load бюджет** = `CLAUDE.md` (925) + `.claude/rules/mindset.md` (1347) + `.claude/rules/workflow.md` (1320) | **3 592** | Постоянная стоимость каждой сессии. `testing.md` исключён — он уже path-scoped `tests/**`. Это **только контент .md**, без harness-обёртки инъекции. |
| **CI-лог** (`gh run view <id> --log`, полный) | **29 477** (~76 KB) | Верхняя граница: `--log-failed` отдаёт только упавший шаг — кратно меньше. |
| **MCP-контекст** (deferred name-list) | **457** (54 имени) | **Схемы MCP НЕ грузятся** — в этом harness MCP-инструменты `deferred`: виден только список имён, схема тянется по запросу через `ToolSearch`. См. вывод C ниже. |

| Категория | Статус |
|---|---|

### Якоря для сравнения

- **1 subagent-спавн = 22 803 токена** — единица «дорого».
- **always-load бюджет = 3 592 токена** ≈ **16 %** одного спавна и ничтожно против типичной рабочей
  сессии (период `1bc9960c…`, opus-4-8: cache-read 630 817, output 20 212, **$1.41**).

## Вывод (Фаза 1)

**Ни один Фаза-2 кандидат из issue #196 не оправдан данными baseline для немедленного issue.**
Это положительный measurement-first исход, а не пустой: главный заявленный рычаг (`mcp-on-demand`)
уже реализован harness'ом (схемы deferred, 457 токенов вместо полных схем), always-load мал
(3 592 ≈ 16 % одного спавна), а дисциплины по спавнам и CI-логам уже закодированы в
`workflow.md §9` / `implement.md §8`. Самый дорогой дискретный рычаг — **subagent-спавн** (~22.8k);
оптимизировать его стоит **дисциплиной** (не плодить спавны на пограничных планах), а не
инфраструктурой. Любой кандидат пересматривается **только** при пересечении своего порога-триггера
из таблицы.
