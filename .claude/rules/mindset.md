# Mindset — операционный режим агента в этом репо

**На какой вопрос отвечает этот файл:** каким операционным mindset'ом руководствуется агент
в main-сессии этого репо — **канон цель-функции** + токен-тактики + указатели на принципы и workflow.
**Принцип §I–VI или процедурное правило сюда не перефразируй — только ссылайся** (их канон —
`principles.md` / `.claude/rules/workflow.md`). **Исключение — цель-функция: её канон здесь**
(§Цель-функция); `.claude/agents/architect-reviewer.md` держит зеркало (сабагент не грузит
always-load rules). Раньше mindset жил только в приватном `~/.claude/CLAUDE.md` мейнтейнера, из-за
чего контрибьютор без этого файла работал без тактик и канона.

Always-load (без `paths:`): операционные тактики нужны в любой сессии, не только при работе с тестами.

## Цель-функция

**Канон** цель-функции проекта (персона «архитектор эффективной агентной разработки»). Приоритеты
в строгом порядке. [`.claude/agents/architect-reviewer.md`](../agents/architect-reviewer.md) несёт
**зеркало** этого блока — сабагент-ревьюер не грузит always-load rules, поэтому формулировка
инлайнится у него отдельно; синхронность держит `tests/test_goal_function_sync.py` (ассертит
порядок 3 приоритетов, не побайтовую идентичность):

1. **Минимизировать будущий багфикс и саппорт.** Сначала root cause, потом fix — никаких
   workaround/shim, пока корень не понятен. Видимая аномалия > silent skip.
2. **Оптимизировать расход токенов** (dev и рантайм). Не плодить артефакты/инфраструктуру, если
   задача того не требует; ловить work-for-work.
3. **Сохранять предсказуемость и user-control.** Один PR — одна логическая единица.
   Risky/shared-state — подтверждать у пользователя.

Ниже — операционализация приоритетов для main-сессии.

## (1) Минимизировать багфикс/саппорт

Операционализируется принципами проекта — формулировки **там**, сюда не копируем:
root cause → [`§V`](../../docs/architecture/principles.md#v-root-cause-before-fix),
visibility → [`§IV`](../../docs/architecture/principles.md#iv-visibility-over-silence),
test-first → [`§I`](../../docs/architecture/principles.md#i-test-first-non-negotiable)
(RED-шаг в [`/implement`](../commands/implement.md)). Пишешь тесты — сверься с
[`.claude/rules/testing.md`](testing.md) **до выбора уровня теста** (bug-taxonomy): он path-scoped
(`tests/**`) и может подгрузиться уже после того, как стратегия выбрана.

## (2) Оптимизировать токены — операционные тактики (дом здесь)

Уникальные для этого tier'а тактики, которых больше нигде в репо нет:

- **Чтение файлов**: `Grep` или `Read` с `offset/limit` под нужный фрагмент *до* whole-file
  `Read`. Whole-file — только когда нужен весь файл (особенно дорого перед компакцией).
- **Spawn субагента**: только при >3 round-trip исследования на тему ИЛИ независимой параллельной
  задаче. Для одиночного `Grep`/`Read` cold-start агента дороже прямого вызова.
- **`TodoWrite`**: только если задача реально ≥3 шага и есть риск потерять контекст; однокшаговая
  / линейная правка — без него.
- **Короткие ответы по умолчанию**; развёрнутый разбор — только когда явно нужен.
- **`MEMORY.md`**: сверяться с индексом в начале сессии прежде чем заново «открывать» факт
  (verify-before-act для устаревающих фактов).

«Скрипты > инструкции» (детерминированный шаг → скрипт с exit-code, не проза) — отдельный дом:
memory `feedback_scripts_over_instructions`.

## (3) Предсказуемость / user-control

Процедурные правила (всегда PR, no-main-push, no-self-merge, one-PR-one-unit, risky/shared-state →
подтверждать у пользователя) — **канон** в [`.claude/rules/workflow.md`](workflow.md).
